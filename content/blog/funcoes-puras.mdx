## O Que É Uma Função Pura?

Uma função é **pura** quando:

1. **Determinística**: Mesmos inputs → Mesmos outputs (sempre)
2. **Sem side effects**: Não modifica estado externo
3. **Sem dependências ocultas**: Apenas dos parâmetros

### Exemplo Simples

✅ **Pura**:
```javascript
const add = (a, b) => a + b

add(2, 3)  // Sempre 5
add(2, 3)  // Sempre 5
```

❌ **Impura**:
```javascript
let total = 0

const addToTotal = (value) => {
  total += value  // Side effect!
  return total
}

addToTotal(5)  // 5
addToTotal(5)  // 10 (resultado diferente!)
```

## Por Que Funções Puras São Melhores?

### 1. Testabilidade Trivial

**Função pura**:
```javascript
const calculate_discount = (price, percentage) => {
  return price * (1 - percentage / 100)
}

// Teste
expect(calculate_discount(100, 10)).toBe(90)
```

Sem mocks, sem setup, sem teardown.

**Função impura** (depende de DB):
```javascript
const calculate_discount = async (user_id, price) => {
  const user = await db.users.findOne(user_id)  // Side effect!
  const discount = user.discount_percentage
  return price * (1 - discount / 100)
}

// Teste complexo
mock_db.users.findOne.returns({ discount_percentage: 10 })
// Setup do mock, teardown, etc
```

### 2. Debugging Previsível

**Pura**: Stack trace mostra exatamente o problema.

```javascript
const calculate_total = (items) => {
  return items.reduce((sum, item) => sum + item.price, 0)
}

calculate_total([{ price: 10 }, { price: null }])
// Error: Cannot read property 'price'
// ↑ Sabemos exatamente onde está o problema
```

**Impura**: Erro pode vir de qualquer lugar.

```javascript
let global_discount = 0.1

const calculate_total = (items) => {
  // Depende de global_discount (modificado em outro lugar)
  const subtotal = items.reduce((sum, item) => sum + item.price, 0)
  return subtotal * (1 - global_discount)
}
```

### 3. Refactoring Seguro

Funções puras podem ser movidas, renomeadas, extraídas sem medo:

```javascript
// Antes
const process_order = (order) => {
  const total = calculate_total(order.items)
  const tax = calculate_tax(total)
  return total + tax
}

// Depois (extraindo lógica)
const calculate_total_with_tax = (items) => {
  const total = calculate_total(items)
  const tax = calculate_tax(total)
  return total + tax
}

const process_order = (order) => {
  return calculate_total_with_tax(order.items)
}
```

Sem quebrar nada.

### 4. Paralelização Automática

Funções puras podem rodar em paralelo sem race conditions:

```javascript
const items = [1, 2, 3, 4, 5]

// Seguro paralelizar
const doubled = items.map(x => x * 2)

// Em Python com multiprocessing
results = pool.map(pure_function, data)
```

## Como Escrever Funções Puras?

### 1. Evite mutação de argumentos

❌ **Evite**:
```javascript
const add_item = (cart, item) => {
  cart.items.push(item)  // Mutação!
  return cart
}
```

✅ **Prefira**:
```javascript
const add_item = (cart, item) => {
  return {
    ...cart,
    items: [...cart.items, item]
  }
}
```

### 2. Isole side effects

❌ **Evite**:
```javascript
const save_user = (user) => {
  validate(user)
  db.save(user)  // Side effect misturado
  return user
}
```

✅ **Separe**:
```javascript
// Pura
const validate_user = (user) => {
  if (!user.email) return { valid: false, error: 'No email' }
  if (!user.name) return { valid: false, error: 'No name' }
  return { valid: true }
}

// Impura (mas isolada)
const save_user_to_db = (user) => {
  return db.save(user)
}

// Orquestração
const create_user = (user) => {
  const validation = validate_user(user)
  
  if (!validation.valid) {
    return error(validation.error)
  }
  
  return save_user_to_db(user)
}
```

### 3. Injete dependências

❌ **Evite** (depende de global):
```javascript
const send_notification = (user, message) => {
  email_service.send(user.email, message)  // Dependência oculta
}
```

✅ **Injete**:
```javascript
const send_notification = (email_service, user, message) => {
  return email_service.send(user.email, message)
}

// Uso
send_notification(real_email_service, user, 'Hello')

// Teste
send_notification(mock_email_service, user, 'Hello')
```

## Exceções Pragmáticas

Nem tudo pode ser puro. **Isole o impuro**:

### Estrutura de camadas

```
Pure Core (90% do código)
  ↕
Side Effects Layer (10%)
  ↕
External World (DB, API, FS)
```

**Exemplo**:

```javascript
// Pure core
const calculate_order = (items, discount_code) => {
  const subtotal = sum_items(items)
  const discount = apply_discount(subtotal, discount_code)
  const tax = calculate_tax(subtotal - discount)
  return { subtotal, discount, tax, total: subtotal - discount + tax }
}

// Side effects layer
const fetch_discount_code = async (code) => {
  return await db.discounts.findOne(code)
}

const save_order = async (order) => {
  return await db.orders.insert(order)
}

// Orchestration (impure)
const process_order = async (items, discount_code) => {
  const discount = await fetch_discount_code(discount_code)
  const order = calculate_order(items, discount)  // Pure!
  await save_order(order)
  return order
}
```

## Impacto em Código Real

Em refactor de um sistema de 10k+ LOC para funções puras:

- **Cobertura de testes**: 45% → 92%
- **Bugs em produção**: -67% (6 meses)
- **Tempo de code review**: -40%
- **Tempo de onboarding**: -50% (novo dev consegue entender funções isoladas)

## Checklist: Minha Função é Pura?

- [ ] Retorna o mesmo output para mesmos inputs?
- [ ] Não modifica argumentos?
- [ ] Não acessa variáveis globais?
- [ ] Não faz I/O (DB, API, FS, console)?
- [ ] Não usa `Date.now()`, `Math.random()`, etc?
- [ ] Pode ser testada sem mocks?

Se **todas** as respostas forem SIM → Pura ✅

## Conclusão

Funções puras são:
- Fáceis de testar
- Fáceis de debugar
- Fáceis de refatorar
- Fáceis de paralelizar

**Regra prática**: Torne 90% das suas funções puras. Isole os 10% impuros.
