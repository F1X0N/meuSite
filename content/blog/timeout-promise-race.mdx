## O cenário concreto

Chamada para LLM com prompt grande. Provider está sobrecarregado, lento ou simplesmente engasgou. Response demora. Sua aplicação espera. Caller upstream também espera.

Sem timeout explícito, três coisas más acontecem:

1. **Backpressure** — sua aplicação fica com requests acumulados, eventualmente esgota pool de conexões.
2. **Sintoma errado no monitoramento** — não é erro, é latência alta. Dashboards de erro não disparam.
3. **Caller desiste** mas sua chamada continua executando, gastando tokens de uma resposta que ninguém vai ler.

A solução é simples e estabelecida: **deadline explícito** combinado com `Promise.race`.

## A mecânica

Duas promises competindo:

- **Operação real** — a chamada para o provider.
- **Timer** — promise que resolve com erro depois de N segundos.

`Promise.race([operacao, timer])` resolve com a primeira que terminar. Se a operação foi mais rápida, retorna o resultado. Se o timer foi mais rápido, lança erro de timeout.

Em pseudo-código:

```
const operacao = client.chat.completions.create({...})
const timer = new Promise((_, reject) =>
  setTimeout(() => reject(new TimeoutError()), 15_000)
)

const result = await Promise.race([operacao, timer])
```

Com 15 segundos de teto, sua aplicação tem garantia de que nenhuma chamada vai segurar request por mais que isso.

## O detalhe que muita implementação esquece

`Promise.race` resolve uma das promises, mas a outra **continua executando**. A operação real não é cancelada quando o timer ganha — ela só não tem mais alguém esperando.

Isso significa duas coisas:

1. **Recursos continuam alocados** durante o tempo restante da operação. Se o provider responder em 30s mesmo após o timeout, sua aplicação ainda paga essa janela em conexão e memória.
2. **Tokens são gastos** mesmo após o timeout. A chamada terminou no servidor do provider, e ele cobrou.

Para cancelar de verdade, é preciso **AbortController**:

```
const controller = new AbortController()
const operacao = client.chat.completions.create({
  ...,
  signal: controller.signal,
})
const timer = new Promise((_, reject) =>
  setTimeout(() => {
    controller.abort()
    reject(new TimeoutError())
  }, 15_000)
)
```

Agora o timer aborta ativamente a operação. Provider recebe sinal de cancelamento (quando suportado), não cobra pelo restante, conexão é fechada.

## Quanto tempo é razoável?

Depende da tarefa.

- **Classificação** (input curto, output curto): 5-10 segundos. Modelo mini responde em 1-2s; 5s já é avisos de problema.
- **Extração estruturada** (input médio, output médio): 15-30 segundos.
- **Geração longa** (output grande): 30-60 segundos. Acima disso, considere streaming em vez de batch.
- **Processamento async** (worker drenando fila): pode ser maior, mas ainda finito. 2-5 minutos por chamada.

A regra prática: **timeout = 2x latência P95 observada em condições normais**. Vai cortar 5% no pior dia, mas evita hang de 10x quando o provider tem incidente.

## Timeout != circuit breaker

Conceitos relacionados, soluções diferentes.

**Timeout** protege uma chamada individual. Aborta cedo. Caller que retentar provavelmente vai sofrer o mesmo timeout.

**Circuit breaker** protege contra padrão. Detecta que múltiplas chamadas seguidas estão falhando, e **recusa novas chamadas por um tempo** sem nem tentar. Dá tempo para o serviço alvo se recuperar.

Em sistema com volume baixo a médio, timeout sozinho geralmente basta. Circuit breaker entra quando você tem dezenas de chamadas concorrentes para o mesmo upstream e precisa proteger sua aplicação de cascateamento.

## Reportando timeout no log

Timeout é evento técnico distinto de erro. Reportar como erro genérico polui o monitoramento.

Convenção que funciona:

- Log nível `warn` (não `error`) com causa `timeout`.
- Inclui o deadline configurado e tempo decorrido (deve bater com o deadline).
- Inclui modelo, tamanho aproximado de input, request_id correlacional.
- Métrica separada: `llm_timeouts_total` por modelo. Subindo = problema upstream.

Sem essa distinção, timeout some no meio de erros 500 e ninguém percebe que o provider está degradando.

## Anti-padrões

- **Timeout sem AbortController** — recursos continuam pendurados, tokens gastos sem ack ao caller.
- **Mesmo timeout para todas as tarefas** — classificação rápida com timeout de 60s mascara modelos lentos.
- **Reportar timeout como erro genérico** — perde visibilidade de degradação.
- **Aumentar timeout pra "dar tempo"** — 99% das vezes o problema não é timeout, é o sistema upstream.

Timeout é mecânica simples mas frequentemente negligenciada em integração com IA. Sem ele, a aplicação fica refém da latência do provider. Com ele, você devolve controle ao caller e evita propagar lentidão.
