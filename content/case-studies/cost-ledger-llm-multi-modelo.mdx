## O problema invisível

IA em produção custa dinheiro real. Cada chamada a um modelo gasta tokens, e tokens custam diferente entre modelos, entre input/output e entre cached/fresh. Sem instrumentação, no fim do mês você recebe uma fatura que não consegue defender — qual feature gastou? Qual cliente puxou esse custo? Faz sentido continuar usando esse modelo nessa tarefa?

A resposta não está no provider. Está no seu sistema, se você modelar isso como **ledger contábil**.

## Ledger não é log

Log de chamada LLM é commodity: provider, modelo, latency, sucesso. Útil pra debugar, inútil pra decidir.

Ledger é diferente. Ledger registra **transação contábil**:

- Quem comprou? (request_id correlacional → trace completo)
- Em nome de qual produto/feature interno?
- Com qual modelo?
- Quantos tokens de cada tipo (input fresh, input cached, output)?
- Custo unitário do modelo no momento da transação?
- Custo derivado total?

Cada chamada gera **uma linha**. Agregação vira analytics — "feature X gastou Y reais este mês", "modelo mini é 40x mais barato que o full nessa tarefa específica", "produto novo está usando custos por requisição 3x acima do orçado".

## O modelo que sustentou a decisão

A tabela de modelos é dado, não código. Para cada modelo suportado:

- preço de input fresh por 1M tokens
- preço de input cached por 1M tokens (alguns modelos cobram menos por contexto repetido)
- preço de output por 1M tokens
- limites operacionais (rate limit, context window)

Quando precificação muda, atualiza a tabela. As chamadas históricas mantêm o preço da época — auditoria não muda retroativamente.

## A arquitetura simples que escala

Counter pattern. Cada chamada incrementa um contador escopado a `(request_id, modelo, produto)`:

1. Antes da chamada: registra início + contexto.
2. Após a chamada: tokens de cada tipo somados, custo derivado pela tabela, persistido junto da entidade alvo (se faz sentido) e na tabela de telemetria.
3. Endpoint analítico: agrega por dimensão (produto, modelo, dia, cliente).

A persistência tem duas camadas:

- **Operacional**: junto da entidade que motivou a chamada (auditoria — "quanto custou processar esta requisição?").
- **Analítica**: tabela achatada para queries de BI, com TTL longo.

## O que isso permite decidir

Decisão 1 — **Modelo certo para a tarefa**. Classificação binária não precisa do modelo top-de-linha. Com ledger, você prova: "mini classifica com mesma acurácia, custa 40x menos. Trocando."

Decisão 2 — **Cache vale a pena**. Ver percentual de input cached por feature mostra onde investir em prompt engineering. Sem dados, é fé.

Decisão 3 — **Pricing de produto novo**. Antes de ofertar feature de IA com preço fixo, simular custo médio com dados reais de adoção esperada. Sem ledger, é chute.

Decisão 4 — **Alerta de anomalia**. Custo médio por requisição salta 3x — feature regrediu (output explodiu, retry agressivo). Sem instrumentação, descobre na fatura.

## Trade-offs honestos

Ledger custa storage e custa overhead de escrita. Para volume baixo, é overengineering. O ponto onde vale a pena é quando:

- Você tem mais de uma feature de IA no mesmo sistema.
- O custo já é visível na fatura (acima de algumas centenas de dólares por mês).
- Decisão de modelo não é mais "qual é o melhor" mas "qual é o adequado por tarefa".

Antes desse ponto, log estruturado com modelo + tokens é suficiente.

## O que não vale a pena fazer

- **Ledger inline na resposta** — incluir custo no payload da API expõe métrica interna. Nunca foi pedido pelo cliente.
- **Cobrar usuário pelo custo bruto** — quase sempre o modelo de pricing precisa de margem e variabilidade. Ledger é input, não output.
- **Repassar precificação do provider direto** — provider muda preço sem aviso. Tabela versionada local protege.

## Princípio operacional

A regra que tornou isso defensável aqui: **toda chamada de LLM deve responder à pergunta "quanto custou e para quem?"**. Antes da chamada é instrumentação. Depois da chamada é prova. Sem prova, IA é caixa preta cara — e caixa preta cara perde para alternativa mensurável.

[Anti-corruption layer entre LLM e domínio](/blog/anti-corruption-layer-llm) é o complemento natural: validação na fronteira impede output bugado de inflar custo. Junto com ledger, você tem visibilidade e proteção.
