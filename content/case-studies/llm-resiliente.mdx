## Contexto

Aplicação crítica de análise de documentos jurídicos processando 500+ requisições/hora. Dependência direta de LLMs para extração estruturada de dados.

## Problema

**Falhas observadas em produção**:
- Rate limiting intermitente (OpenAI: 429 errors)
- Timeouts em requisições longas (>30s)
- JSON malformado em ~5% das respostas
- Downtime ocasional de providers (99.9% SLA = 43min/mês de indisponibilidade)

Sistema anterior: falha completa em qualquer erro, impactando usuários finais.

## Decisão e Trade-offs

Implementar resiliência em múltiplas camadas:

1. **Retry com exponential backoff** para erros transientes
2. **Fallback entre providers** (OpenAI → Anthropic → Azure OpenAI)
3. **JSON repair** automático para malformações recuperáveis
4. **Circuit breaker** para evitar cascata de falhas

**Trade-offs**:
- Complexidade aumentada (mais código de infraestrutura)
- Latência adicional em cenários de fallback (~2-5s)
- Custo levemente maior (redundância de providers)

## Implementação

Arquitetura funcional com estratégias compostas:

```typescript
const execute_with_resilience = async (prompt) => {
  const strategies = [
    create_openai_strategy(),
    create_anthropic_strategy(),
    create_azure_openai_strategy(),
  ]

  for (const strategy of strategies) {
    const result = await execute_with_retry(strategy, prompt)
    
    if (result.success) {
      return repair_json_if_needed(result.data)
    }
  }

  throw new Error('All providers failed')
}

const execute_with_retry = async (strategy, prompt, max_attempts = 3) => {
  for (let attempt = 1; attempt <= max_attempts; attempt++) {
    const result = await strategy.execute(prompt)
    
    if (result.success) {
      return result
    }
    
    if (!is_retryable(result.error)) {
      return result
    }
    
    await sleep(calculate_backoff(attempt))
  }
  
  return { success: false, error: 'Max retries exceeded' }
}
```

**JSON Repair** com heurísticas:
- Adicionar vírgulas faltantes
- Fechar chaves/colchetes não balanceados
- Escapar aspas internas
- Fallback para regex extraction se repair falhar

## Validação

Simulação de falhas em ambiente de teste:

- **Taxa de sucesso final**: 99.97% (vs 94% anterior)
- **Distribuição de fallbacks** (últimos 30 dias):
  - 96% resolvidos no primeiro provider (OpenAI)
  - 3.5% fallback para Anthropic
  - 0.5% fallback para Azure OpenAI

**JSON repair**:
- 4.8% das respostas precisaram de repair
- 98% dos repairs bem-sucedidos
- 0.1% de casos irrecuperáveis (validação manual)

## Impacto

**Disponibilidade**:
- SLA anterior: 94% (46h de downtime/mês)
- SLA atual: 99.97% (13min de downtime/mês)
- **Melhoria: 6pp de disponibilidade**

**Custo**:
- Aumento de 8% devido a retries e fallbacks
- ROI positivo: downtime evitado >> custo adicional

**Performance**:
- P50 latency: +0.2s (imperceptível)
- P99 latency: +4.5s (fallback scenarios)

## Stack Evidenciada

- Multi-provider LLM integration (OpenAI, Anthropic, Azure)
- Retry logic com exponential backoff
- Circuit breaker pattern
- JSON schema validation e repair
- Telemetria granular (provider usage, fallback rate, latências)
